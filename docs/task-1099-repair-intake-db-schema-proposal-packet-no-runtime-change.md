## Repair Intake Drafts

Potential future schema concepts for repair intake drafts:

- draft identity: stable internal draft id generated by the system
- organization and tenant isolation: explicit organization scope on every draft record
- source and intake channel: API, CSV, Excel, web, LINE-scoped channel, phone, assisted report, or future AI call source
- draft status: received, needs_review, ready_for_case, converted, rejected, expired, archived, or equivalent future state
- safe reference fields: external reference id, masked channel reference, import batch reference, and non-sensitive source reference
- timestamps: created, updated, reviewed, converted, rejected, expired, archived
- PII / raw payload policy: keep raw payloads out of customer-visible envelopes, minimize storage, mask sensitive fields, and avoid full phone/address/customer identity in broad read paths
- LINE identity policy: LINE user id must never be treated as a global identity and must remain scoped by organization and LINE channel
- archival posture: prefer explicit retention/archive state before destructive deletion; soft-delete or archival strategy requires separate approval

Candidate scope only. No table is authorized here.

## Draft-to-Case Conversion Records

Potential future schema concepts for draft-to-Case conversion:

- draft id: reference to the repair intake draft
- case id or case reference: reference to the created Case after successful conversion
- conversion status: planned, submitted, succeeded, duplicate_replayed, rejected, failed, or equivalent future state
- idempotency key: scoped key used to detect safe replay
- actor/context reference: actor id, organization scope, role/context reference, and permission decision reference where appropriate
- timestamps: planned, submitted, completed, failed, replayed
- duplicate and conflict handling: record whether a request created a new Case, replayed an existing conversion, or failed closed due to conflict
- customer-visible posture: conversion internals should remain internal unless a future customer-facing contract explicitly authorizes safe fields

Candidate scope only. No conversion table or repository writer is authorized here.

## Idempotency Records

Potential future schema concepts for idempotency:

- idempotency key: caller-supplied or system-derived key, never globally unique without organization/tenant scope
- operation type: draft_to_case_plan, draft_to_case_submit, or equivalent future operation
- organization and tenant scope: included in uniqueness and lookup constraints
- request hash or safe fingerprint: deterministic fingerprint of the minimum necessary request shape, without storing raw sensitive payload
- replay result reference: reference to conversion result, Case reference, or safe outcome envelope
- expiration / retention concept: retention should be bounded and separately approved
- uniqueness concept: organization scope plus operation type plus idempotency key, with request fingerprint conflict detection
- conflict posture: same key with different safe fingerprint should fail closed and avoid accidental duplicate Case creation

Candidate scope only. No idempotency table, index, migration, or repository writer is authorized here.

## Audit Events

Potential future schema concepts for Repair Intake audit events:

- event type: draft_received, draft_reviewed, draft_to_case_planned, draft_to_case_submitted, conversion_succeeded, conversion_replayed, conversion_failed, permission_denied, conflict_detected
- draft id: included when the event relates to a draft
- case id or case reference: included only after a Case exists or is safely referenced
- actor/context: actor id, organization scope, role/context reference, permission outcome, and non-sensitive request context
- decision / outcome: accepted, rejected, replayed, denied, failed, or requires_review
- safe metadata: no token, secret, full phone, full address, raw LINE user id, AI raw payload, or unnecessary customer PII
- timestamps: event creation time and relevant decision time
- customer-visible vs internal-only posture: audit records default to internal-only; customer-visible event summaries require separate explicit API approval

Candidate scope only. No audit persistence table, writer, or API exposure is authorized here.

## Cross-Cutting Constraints

Future schema work must preserve:

- organization isolation on every persisted row
- tenant and SaaS readiness
- permission and audit traceability
- no raw phone, full address, customer PII, token, secret, raw LINE identity, or AI raw payload in customer-visible envelopes
- no LINE identity as global identity
- idempotency uniqueness scoped by organization/tenant and operation type
- transaction boundary covering draft read, Case creation, idempotency record, conversion record, and audit event where appropriate
- retention and privacy posture for drafts, raw import references, idempotency records, and audit events
- one Case / one formal completion report invariant
- human-in-the-loop posture for dispatch and appointment decisions
