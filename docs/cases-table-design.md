# Cases Main Table Design

## Purpose

`cases` is the central case master table for phase 1 repair requests. It stores the repair request identity, current lifecycle state, product and fault details, AI-assisted summary fields, and a light dispatch preview.

It intentionally does not store the current customer profile, attachments, message history, full dispatch workflow, engineer service reports, or billing records. `customer_snapshot` is the narrow exception for historical audit evidence at case creation time.

## Related Table Assumptions

| Table | Assumption |
|---|---|
| `customers` | Has `id uuid primary key`; stores customer name, phone, LINE id, city, and address. |
| `dispatch_units` | Has `id uuid primary key`; stores dispatch unit metadata, priority, enable/disable state, and routing rules. |
| `users` | Has `id uuid primary key`; stores admin or system users for audit attribution. |
| `case_attachments` | Stores R2 object metadata and attachment type, linked by `case_id`. |
| `case_messages` | Stores LINE, website, admin, and AI message history, linked by `case_id`. |
| `audit_logs` | Stores traceable create/update/review/status-change events. |

## Field List

| Field | Type | Nullable | Reason |
|---|---|---:|---|
| `id` | `uuid` | no | Primary key. Uses UUID for distributed-safe identifiers. |
| `case_no` | `text` | no | Human-readable case number returned to customers and searched by admins. Generated by application or database sequence policy. Future recommended format: `TW-YYYYMMDD-000001`. |
| `customer_id` | `uuid` | no | Links to `customers`; avoids duplicating personal profile fields in `cases`. |
| `status` | `text` | no | Phase 1 lifecycle state, with reserved future states allowed by constraint. |
| `priority` | `text` | no | Case priority for urgent cases, VIP handling, future SLA support, and dispatch queue sorting. |
| `warranty_status` | `text` | no | Warranty review state that may later affect dispatch, manufacturer claims, and billing reconciliation. |
| `appointment_status` | `text` | no | Case-level appointment summary for phase 1.5. Full appointment history should live in future `appointments`, not `cases`. |
| `completion_status` | `text` | no | Case-level completion summary for phase 1.5. Full completion records should live in future `field_service_reports`, not `cases`. |
| `source` | `text` | no | Entry channel. Phase 1 uses LINE and website, while phone, WhatsApp, Facebook, Instagram, and email are reserved. |
| `brand` | `text` | no | Required product brand from repair request. |
| `case_type` | `text` | no | Case type, such as repair, installation, maintenance, inspection, return, warranty, or other. Do not use `product_type` as a substitute. |
| `product_type` | `text` | no | Product category/type for triage and dispatch routing. |
| `model_no` | `text` | no | Product model. |
| `serial_no` | `text` | yes | Required in the flow, but nullable while OCR or customer follow-up is pending. |
| `invoice_date` | `date` | yes | Can be entered manually or filled by OCR from invoice photo. |
| `problem_description` | `text` | no | Customer-provided fault description. |
| `preferred_visit_time` | `timestamptz` | yes | Desired on-site visit time when supplied. |
| `service_region` | `text` | yes | Operational service region, such as north, central, south, or a company-defined region. This is not the same as `dispatch_unit_id`. |
| `ai_summary` | `text` | yes | AI-generated case summary for admin review. |
| `ai_classification` | `jsonb` | yes | AI classification output, kept flexible for provider/model changes. |
| `ai_confidence` | `numeric(5,4)` | yes | Confidence score from 0 to 1. This is advisory only and must not be the sole basis for automatic decisions. |
| `ai_suggested_dispatch_unit_id` | `uuid` | yes | AI recommendation only; does not imply final assignment. |
| `ai_ocr_status` | `text` | no | Tracks OCR state for serial/invoice extraction. |
| `dispatch_unit_id` | `uuid` | yes | Current preview/manual selected dispatch unit for phase 1 admin review. |
| `dispatch_assignment_source` | `text` | yes | Indicates whether dispatch preview came from rule, AI, or manual override. |
| `customer_snapshot` | `jsonb` | yes | Historical snapshot of customer data at case creation, such as name, phone, and address. This supports auditability and does not replace `customers`. |
| `metadata` | `jsonb` | yes | Low-risk extensibility area for LINE metadata, migration data, or temporary integration data. Formal long-term fields must not be hidden here. |
| `created_at` | `timestamptz` | no | Creation timestamp. |
| `updated_at` | `timestamptz` | no | Last update timestamp, automatically maintained by trigger on update. |
| `created_by` | `uuid` | yes | User/system attribution for auditability. |
| `updated_by` | `uuid` | yes | Last updater attribution for auditability. |
| `submitted_at` | `timestamptz` | yes | Set when customer submission is complete. |
| `reviewed_at` | `timestamptz` | yes | Set when admin review occurs. |
| `accepted_at` | `timestamptz` | yes | Set when case is accepted. |
| `rejected_at` | `timestamptz` | yes | Set when case is rejected. |
| `cancelled_at` | `timestamptz` | yes | Set when case is cancelled. |
| `scheduled_at` | `timestamptz` | yes | Case-level confirmed appointment time summary. Full appointment history belongs in future `appointments`. |
| `completed_at` | `timestamptz` | yes | Case-level completion time summary. Full on-site service records belong in future `field_service_reports`. |
| `last_customer_message_at` | `timestamptz` | yes | Last customer message time for service queue ordering and timeout monitoring. Conversation history remains in `case_messages`. |
| `last_internal_activity_at` | `timestamptz` | yes | Last internal system, AI, customer service, or admin handling time for queue and stuck-case monitoring. Full activity history remains in audit/messages/future event tables. |
| `deleted_at` | `timestamptz` | yes | Soft-delete marker for archival strategy. |

## Constraint Strategy

Use `CHECK` constraints instead of PostgreSQL enum types for phase 1. The allowed values may expand as dispatch, engineer service, and billing modules come online, and check constraints are easier to alter in early product iterations.

Recommended constrained fields:

- `status`
- `priority`
- `warranty_status`
- `appointment_status`
- `completion_status`
- `case_type`
- `source`
- `problem_description`
- `ai_ocr_status`
- `dispatch_assignment_source`
- `ai_confidence`
- lifecycle timestamp consistency

`case_no` remains `text` in phase 1. It should be generated by the application or a database sequence policy rather than hardcoded into the `cases` table. The recommended future display format is `TW-YYYYMMDD-000001`.

## PostgreSQL SQL

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE cases (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    case_no text NOT NULL,
    customer_id uuid NOT NULL REFERENCES customers(id),

    status text NOT NULL DEFAULT 'draft',
    priority text NOT NULL DEFAULT 'normal',
    warranty_status text NOT NULL DEFAULT 'unknown',
    appointment_status text NOT NULL DEFAULT 'not_required',
    completion_status text NOT NULL DEFAULT 'not_started',
    source text NOT NULL,

    brand text NOT NULL,
    case_type text NOT NULL DEFAULT 'repair',
    product_type text NOT NULL,
    model_no text NOT NULL,
    serial_no text,
    invoice_date date,
    problem_description text NOT NULL,
    preferred_visit_time timestamptz,
    service_region text,

    ai_summary text,
    ai_classification jsonb,
    ai_confidence numeric(5,4),
    ai_suggested_dispatch_unit_id uuid REFERENCES dispatch_units(id),
    ai_ocr_status text NOT NULL DEFAULT 'not_started',

    dispatch_unit_id uuid REFERENCES dispatch_units(id),
    dispatch_assignment_source text,

    customer_snapshot jsonb,
    metadata jsonb,

    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    created_by uuid REFERENCES users(id),
    updated_by uuid REFERENCES users(id),
    submitted_at timestamptz,
    reviewed_at timestamptz,
    accepted_at timestamptz,
    rejected_at timestamptz,
    cancelled_at timestamptz,
    scheduled_at timestamptz,
    completed_at timestamptz,
    last_customer_message_at timestamptz,
    last_internal_activity_at timestamptz,
    deleted_at timestamptz,

    CONSTRAINT cases_case_no_unique UNIQUE (case_no),
    CONSTRAINT cases_status_check CHECK (
        status IN (
            'draft',
            'pending_customer',
            'submitted',
            'reviewing',
            'accepted',
            'rejected',
            'cancelled',
            'dispatch_pending',
            'assigned',
            'scheduled',
            'on_site',
            'completed',
            'closed'
        )
    ),
    CONSTRAINT cases_priority_check CHECK (
        priority IN ('low', 'normal', 'high', 'urgent', 'vip')
    ),
    CONSTRAINT cases_warranty_status_check CHECK (
        warranty_status IN (
            'unknown',
            'pending_review',
            'in_warranty',
            'out_of_warranty'
        )
    ),
    CONSTRAINT cases_appointment_status_check CHECK (
        appointment_status IN (
            'not_required',
            'pending',
            'requested',
            'confirmed',
            'reschedule_requested',
            'cancelled'
        )
    ),
    CONSTRAINT cases_completion_status_check CHECK (
        completion_status IN (
            'not_started',
            'in_progress',
            'completed',
            'customer_confirmed',
            'disputed'
        )
    ),
    CONSTRAINT cases_case_type_check CHECK (
        case_type IN (
            'repair',
            'installation',
            'maintenance',
            'inspection',
            'return',
            'warranty',
            'other'
        )
    ),
    CONSTRAINT cases_source_check CHECK (
        source IN (
            'line',
            'website',
            'admin',
            'api',
            'migration',
            'phone',
            'whatsapp',
            'facebook',
            'instagram',
            'email'
        )
    ),
    CONSTRAINT cases_problem_description_not_blank_check CHECK (
        length(trim(problem_description)) > 0
    ),
    CONSTRAINT cases_ai_ocr_status_check CHECK (
        ai_ocr_status IN (
            'not_started',
            'pending',
            'processing',
            'partial',
            'completed',
            'failed',
            'manual_review'
        )
    ),
    CONSTRAINT cases_dispatch_assignment_source_check CHECK (
        dispatch_assignment_source IS NULL
        OR dispatch_assignment_source IN ('rule', 'ai', 'manual')
    ),
    CONSTRAINT cases_ai_confidence_check CHECK (
        ai_confidence IS NULL OR (ai_confidence >= 0 AND ai_confidence <= 1)
    ),
    CONSTRAINT cases_deleted_after_created_check CHECK (
        deleted_at IS NULL OR deleted_at >= created_at
    )
);

CREATE FUNCTION set_cases_updated_at()
RETURNS trigger AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_cases_set_updated_at
BEFORE UPDATE ON cases
FOR EACH ROW
EXECUTE FUNCTION set_cases_updated_at();

CREATE INDEX idx_cases_customer_id ON cases(customer_id);
CREATE INDEX idx_cases_status ON cases(status) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_status_priority_submitted_at ON cases(status, priority, submitted_at DESC) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_warranty_status ON cases(warranty_status) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_source ON cases(source) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_dispatch_unit_id ON cases(dispatch_unit_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_ai_suggested_dispatch_unit_id ON cases(ai_suggested_dispatch_unit_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_created_at ON cases(created_at DESC);
CREATE INDEX idx_cases_submitted_at ON cases(submitted_at DESC) WHERE submitted_at IS NOT NULL;
CREATE INDEX idx_cases_scheduled_at ON cases(scheduled_at DESC) WHERE scheduled_at IS NOT NULL;
CREATE INDEX idx_cases_completed_at ON cases(completed_at DESC) WHERE completed_at IS NOT NULL;
CREATE INDEX idx_cases_last_customer_message_at ON cases(last_customer_message_at DESC) WHERE last_customer_message_at IS NOT NULL;
CREATE INDEX idx_cases_last_internal_activity_at ON cases(last_internal_activity_at DESC) WHERE last_internal_activity_at IS NOT NULL;
CREATE INDEX idx_cases_case_type ON cases(case_type) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_service_region ON cases(service_region) WHERE service_region IS NOT NULL AND deleted_at IS NULL;
CREATE INDEX idx_cases_brand_product_type ON cases(brand, product_type) WHERE deleted_at IS NULL;
CREATE INDEX idx_cases_search_text ON cases USING gin (
    to_tsvector(
        'simple',
        coalesce(case_no, '') || ' ' ||
        coalesce(brand, '') || ' ' ||
        coalesce(product_type, '') || ' ' ||
        coalesce(model_no, '') || ' ' ||
        coalesce(serial_no, '') || ' ' ||
        coalesce(problem_description, '')
    )
);
```

## Usage Notes

`case_type` describes what kind of work the case represents. It is separate from `product_type`: for example, `product_type` may be television while `case_type` may be installation, repair, or maintenance.

`service_region` is an operational service area such as north, central, south, or another company-defined region. It is not the same as `dispatch_unit_id`; the latter points to a dispatch unit.

`customer_snapshot` should only store the minimum necessary customer data at case creation time, such as `name`, `mobile`, `tel`, `city`, and `address`. Do not store a complete customer object, because that would duplicate too much personal data.

`metadata` must not be used as the basis for formal core workflow. Do not hide long-term official fields in `metadata`. Once a value becomes part of the core business process, promote it to an explicit column or a related table.

`ai_confidence` is an advisory value only. It must not be the sole basis for automatic decisions, because different AI providers may define confidence differently.

## What Should Not Be Placed In `cases`

- Customer name, mobile, telephone, LINE user id, city, or address. Use `customers`.
- Current customer profile should remain in `customers`; `customer_snapshot` is only a historical audit snapshot captured at case creation.
- Photos, invoice images, nameplate images, or R2 object keys. Use `case_attachments`.
- LINE, website, admin, or AI conversation records. Use `case_messages`.
- Full dispatch scheduling, engineer assignment history, route planning, or SLA workflow. Use future `dispatches` and `engineers`.
- On-site repair details, signatures, engineer photos, parts used, or service outcomes. Use future `field_service_reports`, `service_photos`, and related tables.
- Billing settlement details, manufacturer claims, engineer settlements, or vendor reconciliation details. Use future billing tables.
- Raw AI prompts, model responses, token usage, or vendor request logs. Use `ai_logs`.
- Long-term formal fields should not be hidden in `metadata`; promote them to explicit columns or related tables once they become core behavior.

## Future Extension Notes

Dispatch should introduce `dispatches` as the operational assignment table. `cases.dispatch_unit_id` remains a phase 1 preview/current summary field and should not replace assignment history.

Field service should introduce `field_service_reports`, `service_photos`, and signatures linked through dispatch or case IDs. Completion state can update `cases.status`, but detailed engineer workflow belongs outside `cases`.

`preferred_visit_time` stays as a single `timestamptz` in phase 1. If customers later need to submit a time range such as 14:00-18:00, this can be expanded to `preferred_visit_start` and `preferred_visit_end`.

Billing should introduce `billing_records`, `manufacturer_claims`, `engineer_settlements`, and `vendor_settlements`. `cases` may be referenced by those tables, but it should not store settlement amounts, reconciliation status, or invoice line details.

Phase 1.5 may use `appointment_status` and `completion_status` as case-level summaries only. Full appointment records should live in future `appointments`; complete engineer assignment history should live in future `dispatches` or `dispatch_assignments`; on-site records should live in future `field_service_reports`.

Phase 1.5 may also use `scheduled_at`, `completed_at`, `last_customer_message_at`, and `last_internal_activity_at` as operational summary timestamps. These are not replacements for full workflow records: appointments belong in `appointments`; dispatch history belongs in `dispatches` or `dispatch_assignments`; field service records belong in `field_service_reports`; billing belongs in `billing_records` and `settlements`; notification records belong in `notification_logs` and `notification_preferences`.
